## 0. 최우선 원칙 (절대선)

### 0.1 False Positive는 사용자 이탈 직행

이 도구의 존재 이유는 위험 감지인데, 잘못 잡으면 사용자가 즉시 uninstall한다.
다음 원칙을 어기면 FP가 발생한다:

- **컨텍스트 없이 패턴만 매칭** 금지: `@RequireTenant()`가 있는 메서드는 `missing-tenant-filter` skip 필수
- **검사 제외 마커 미지원** 금지: 모든 패턴에 `// nestjs-leak-guard-disable-next-line` 지원 필수
- **테스트 픽스처 없는 패턴 배포** 금지

### 0.2 명확한 수정 제안 없는 경고 금지

"이 코드가 위험합니다" 메시지만 내면 사용자가 무엇을 해야 할지 모른다.
반드시 `suggestion` 필드에 **어떻게 고쳐야 하는지** 코드 수준으로 제시.

## 4. 설계 결정 기록

### 4.1 ts-morph 채택 — TypeScript Compiler API 직접 사용 vs ts-morph

**결정**: ts-morph 사용.

**이유**: TypeScript Compiler API는 low-level이라 AST 탐색 코드가 장황해짐.
ts-morph는 같은 API를 래핑해서 코드 가독성을 크게 높임.
nestjs-leak-guard는 패턴 개수가 늘어날수록 AST 코드량이 커지므로 가독성 우선.

**트레이드오프**: ts-morph 버전 의존성 생김. peer dep 정책에 ts-morph 버전 명시 필요.

### 4.2 ESLint 플러그인 vs 독립 CLI — 우선순위

**결정**: 독립 CLI 먼저, ESLint 플러그인은 3단계.

**이유**: ESLint 플러그인은 ESLint 버전 호환성 관리 부담이 큼 (ESLint 8 vs 9 API 차이).
독립 CLI는 의존성 최소화, 바로 쓸 수 있음. 차별화도 독립 CLI 쪽이 더 명확.
